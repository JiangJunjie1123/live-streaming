# 音视频项目 VideoRecoder 部分阶段性

## 项目概述

VideoRecoder 是一款基于 Qt 的桌面视频录制应用程序，支持摄像头捕获、桌面录制、画中画合成以及视频编码保存功能。本文档记录当前阶段的技术选型、架构设计和实现细节。

## 问题总结：

### Qt 信号槽

**问题：Qt 信号槽的原理是什么？**

1. **实现原理**
   
   - 基于元对象系统（Meta-Object System）
   - MOC（Meta-Object Compiler）预处理信号槽声明
   - 生成 `moc_*.cpp` 文件包含连接信息
   - 运行时通过 `QObject::connect` 建立连接

2. **连接类型（Qt::ConnectionType）**
   
   ```cpp
   Qt::AutoConnection       // 自动判断（同线程直接调用，跨线程队列）
   Qt::DirectConnection     // 直接调用（同一线程）
   Qt::QueuedConnection     // 队列调用（跨线程安全）
   Qt::BlockingQueuedConnection // 阻塞等待（跨线程同步）
   ```

3. **本项目中的应用**
   
   - 视频帧数据通过 `QueuedConnection` 跨线程传递
   - UI 更新使用 `DirectConnection` 保证实时性
   - 避免在主线程进行耗时操作（编码在子线程）

**为什么不用回调函数？**

- 信号槽是类型安全的（编译期检查）
- 支持多对多连接
- 自动处理跨线程通信
- 松耦合设计，模块独立

### FFmpeg 编码流程详解

**问题：FFmpeg 编码视频的完整流程是什么？**

**回答要点：**

1. **核心结构体关系**
   
   ```
   AVFormatContext (格式上下文)
       ├── AVStream[] (流数组)
       │       └── AVCodecContext (编码器上下文)
       │               └── AVCodec (编码器)
       └── AVIOContext (IO上下文)
   
   AVFrame (原始帧) → 编码 → AVPacket (压缩包)
   ```

2. **编码流程代码级讲解**
   
   ```cpp
   // 1. 分配格式上下文
   avformat_alloc_output_context2(&oc, NULL, NULL, filename);
   
   // 2. 创建视频流
   AVStream *st = avformat_new_stream(oc, NULL);
   
   // 3. 配置编码器参数
   AVCodecContext *c = avcodec_alloc_context3(codec);
   c->width = 1920; c->height = 1080;
   c->time_base = {1, 25};  // 时基
   c->framerate = {25, 1};  // 帧率
   c->pix_fmt = AV_PIX_FMT_YUV420P;
   
   // 4. 打开编码器
   avcodec_open2(c, codec, &opts);
   
   // 5. 写入文件头
   avformat_write_header(oc, NULL);
   
   // 6. 编码循环
   while (has_frame) {
       // 填充 AVFrame
       frame->pts = calculate_pts();
   
       // 发送帧给编码器
       avcodec_send_frame(c, frame);
   
       // 接收编码后的包
       while (avcodec_receive_packet(c, &pkt) == 0) {
           av_interleaved_write_frame(oc, &pkt);
       }
   }
   
   // 7. 刷新编码器
   avcodec_send_frame(c, NULL);  // 发送空帧刷新
   
   // 8. 写入文件尾并关闭
   av_write_trailer(oc);
   avcodec_free_context(&c);
   avformat_free_context(oc);
   ```

3. **PTS/DTS 时间戳详解**
   
   - **PTS** (Presentation Time Stamp): 显示时间戳，决定帧何时显示
   - **DTS** (Decoding Time Stamp): 解码时间戳，决定帧何时解码
   - **time_base**: 时间基，表示时间单位（如 {1, 1000} 表示毫秒）
   - **计算关系**: `time_in_seconds = pts * time_base.num / time_base.den`

4. **为什么用 YUV420P？**
   
   - 人眼对亮度敏感，对色度不敏感
   - YUV420P 是 4:2:0 采样，色度分量分辨率减半
   - 相比 RGB24 节省 50% 空间
   - H.264 编码器原生支持

### 多线程与线程安全

**问题：项目中如何处理多线程问题？**

**回答要点：**

1. **线程划分**
   
   ```
   主线程（GUI线程）
   ├── 事件循环处理用户交互
   ├── 接收显示信号更新界面
   └── 不处理耗时操作
   
   工作线程（SaveVideoFileThread继承QThread）
   ├── 视频采集（PicInPic_Read）
   ├── 图像处理与编码
   └── 文件写入
   ```

2. **线程通信方式**
   
   - 信号槽（自动队列连接）
   - 共享数据加锁（QMutex）
   - 原子变量（QAtomicInt）

3. **本项目中的线程安全设计**
   
   ```cpp
   // 录制状态使用原子标志
   QAtomicInt m_isRecording;
   
   // 共享队列加锁保护
   QMutex m_frameQueueMutex;
   QList<QImage> m_frameQueue;
   
   // FFmpeg 上下文只在单线程访问
   // 不需要额外加锁
   ```

4. **死锁避免策略**
   
   - 锁的粒度尽量小
   - 避免在持锁时调用外部代码
   - 使用 QMutexLocker 自动释放
   - 信号槽天然线程安全

### 内存管理细节

**问题：FFmpeg 的内存如何管理？有哪些坑？**

**回答要点：**

1. **FFmpeg 内存分配模式**
   
   ```cpp
   // AVFrame 的内存结构
   AVFrame *frame = av_frame_alloc();  // 分配结构体本身
   av_frame_get_buffer(frame, 0);      // 分配内部数据缓冲区
   
   // 释放时必须用对应函数
   av_frame_free(&frame);  // 自动释放内部缓冲区
   ```

2. **常见内存泄漏点**
   
   - `av_packet_unref()` 忘记调用（复用 packet 时）
   - `av_frame_unref()` 忘记调用
   - `avformat_open_input()` 后没有 `avformat_close_input()`
   - `sws_getContext()` 后没有 `sws_freeContext()`

3. **本项目踩过的坑**
   
   ```cpp
   // 错误：重复释放
   av_free(frame->data[0]);  // ❌ data[0] 属于 AVBufferRef 管理
   av_frame_free(&frame);     // 这会导致 double free
   
   // 正确：只释放 frame
   av_frame_free(&frame);     // ✓ 自动处理内部缓冲区
   ```

4. **智能指针封装建议**
   
   ```cpp
   // 使用 RAII 封装
   struct AVFrameDeleter {
       void operator()(AVFrame* frame) {
           av_frame_free(&frame);
       }
   };
   using AVFramePtr = std::unique_ptr<AVFrame, AVFrameDeleter>;
   
   // 使用
   AVFramePtr frame(av_frame_alloc());
   // 自动释放，无需手动调用
   ```

### 性能优化经验

**问提：项目中做了哪些性能优化？**

**回答要点：**

1. **编码性能优化**
   
   - **预设参数**: `superfast` 牺牲压缩率换取速度
   - **零延迟模式**: `zerolatency` 适用于实时场景
   - **多线程编码**: `thread_count = 4` 利用多核
   - **GOP 设置**: 12 帧一个关键帧，平衡压缩和随机访问

2. **内存优化**
   
   - 使用 `av_frame_make_writable()` 避免不必要的拷贝
   - 复用 AVPacket 结构体（先 unref 再复用）
   - 图像缩放使用 `SWS_FAST_BILINEAR` 快速算法

3. **CPU 优化**
   
   - 颜色空间转换使用 SIMD 优化（FFmpeg 内部已实现）
   - 避免在主线程做图像处理
   - 使用 `QImage::bits()` 直接访问像素数据，避免深拷贝

4. **IO 优化**
   
   - 使用 `av_interleaved_write_frame()` 自动缓冲排序
   - 大文件使用内存映射（可选优化）

### 视频格式与协议

**问题：FLV 格式有什么特点？为什么选它？**

**回答要点：**

1. **FLV 格式特点**
   
   - Flash Video 格式，简单轻量
   - 头部信息小，适合流式传输
   - 支持 H.264 + AAC（现代 FLV）
   - 与 RTMP 协议天然兼容

2. **与其他格式对比**
   
   | 格式  | 优点           | 缺点          | 适用场景   |
   | --- | ------------ | ----------- | ------ |
   | FLV | 简单、流式、RTMP兼容 | 老旧、功能有限     | 直播、录制  |
   | MP4 | 通用、支持丰富      | 头部在尾部，不适合流式 | 本地播放   |
   | MKV | 功能强大、开源      | 复杂、兼容性一般    | 高质量存储  |
   | AVI | 简单兼容         | 老旧、压缩率低     | 兼容老旧设备 |

3. **RTMP 推流准备**
   
   - FLV 是 RTMP 的容器格式
   - 只需将 `avio_open()` 改为 `avio_open_dyn_buf()` 或直接推送到 rtmp:// URL
   - 需要处理网络缓冲和重连逻辑

### 2.3 音频采集模块 (Audio Capture Module) 【2026.4.8 】

#### 技术选型：Qt Multimedia (QAudioInput)

**为何选择 Qt Multimedia：**

- Qt 原生支持，无需额外依赖
- API 简单易用，通过 QAudioInput 直接采集
- 使用定时器控制采集节奏，流程清晰
- 相比 FFmpeg 直接采集设备，故障率更低

**核心功能：**

```cpp
// 音频格式配置
QAudioFormat format;
format.setSampleRate(44100);        // 采样率 44.1kHz
format.setChannelCount(2);          // 双声道
format.setSampleSize(16);           // 16位采样
format.setCodec("audio/pcm");       // PCM 格式
format.setByteOrder(QAudioFormat::LittleEndian);
format.setSampleType(QAudioFormat::UnSignedInt);

// 创建音频输入对象
QAudioInput* audio_in = new QAudioInput(format, this);
QIODevice *myBuffer_in = audio_in->start();
```

**技术实现要点：**

1. **音频参数配置**
   
   - 采样率：44100 Hz（CD 音质标准）
   - 声道数：2（立体声）
   - 采样位数：16 bit
   - 数据格式：PCM 小端模式

2. **重采样转换**
   
   - 输入格式：S16 (signed 16-bit PCM)
   - 输出格式：FLTP (float planar，AAC 编码所需)
   - 使用 FFmpeg libswresample 进行转换
   - 转换公式：`swr_convert(swr, out_buffer, ...)`

3. **帧大小计算**
   
   - AAC 规定每帧 1024 个采样点
   - 一帧数据量：`1024 * 2声道 * 4字节 = 8192 字节`
   - 采样间隔：20ms（根据 44100Hz / 1024 ≈ 43fps 计算）

**编码流程：**

```
QAudioInput 采集 (PCM S16)
    ↓
swr_convert (S16 → FLTP)
    ↓
SIG_sendAudioFrameData (信号发射)
    ↓
SaveVideoFileThread::slot_writeAudioFrameData
    ↓
avcodec_encode_audio2 (AAC 编码)
    ↓
av_interleaved_write_frame (写入文件)
```

**关键代码片段：**

```cpp
// 重采样上下文初始化
swr = swr_alloc_set_opts(nullptr,
    AV_CH_LAYOUT_STEREO, AV_SAMPLE_FMT_FLTP, 44100,  // 输出
    AV_CH_LAYOUT_STEREO, AV_SAMPLE_FMT_S16, 44100,   // 输入
    0, nullptr);
swr_init(swr);

// 音频数据转换
uint8_t* out_buffer[2];  // 左右声道分开存储
swr_convert(swr, out_buffer, samples,
            (const uint8_t**)&in_buffer, samples);
```

**优势：**

- 与 Qt 信号槽机制无缝集成
- 跨平台兼容性好（Windows/Linux/macOS）
- 采集和编码分离，便于独立调试
- 支持动态启停，响应及时

**注意事项：**

- 需要添加 `QT += multimedia` 到 .pro 文件
- 音频设备可能被其他应用占用，需要错误处理
- PCM 数据需要及时读取，避免缓冲区溢出

### 2.4 视频编码模块 (Video Encoding Module)

#### 技术选型：FFmpeg 4.2.2

**为何选择 FFmpeg：**

- 业界标准的音视频处理框架
- 支持 H.264 硬件加速编码
- 丰富的封装格式支持（FLV、MP4、AVI 等）
- 支持 RTMP 推流（为未来直播功能预留）

**核心组件：**

| FFmpeg 模块   | 用途     | 关键 API                                    |
| ----------- | ------ | ----------------------------------------- |
| libavcodec  | 视频编码   | avcodec\_encode\_video2                   |
| libavformat | 格式封装   | avformat\_write\_header, av\_write\_frame |
| libswscale  | 颜色空间转换 | sws\_scale (RGB→YUV420P)                  |
| libavutil   | 工具函数   | av\_frame\_alloc, av\_packet\_alloc       |

**编码流程：**

```
QImage (RGB24) 
    ↓
sws_scale → YUV420P
    ↓
avcodec_encode_video2 → H.264 数据包
    ↓
av_interleaved_write_frame → 写入文件
```

**关键技术决策：**

1. **编码器选择：H.264 (libx264)**
   - 原因：压缩率高、兼容性好、硬件支持广泛
   - 预设参数：`superfast` + `zerolatency`（低延迟实时编码）
2. **颜色空间：YUV420P**
   - 原因：H.264 标准支持、压缩效率高
   - 转换：通过 sws\_scale 从 RGB24 转换
3. **PTS 计算策略**
   - 早期：简单递增 `next_pts++`
   - 改进：基于时间戳计算 `(elapsedMs * frame_rate) / 1000`
   - 优势：确保播放速度与录制时长一致
4. **帧率控制**
   - 编码器帧率：`c->framerate = {25, 1}`
   - 流帧率：`st->avg_frame_rate = {25, 1}`
   - 时基：`time_base = {1, 25}`

**优势：**

- 编码效率高，CPU 占用可控
- 输出文件体积小，画质损失少
- 为未来直播功能预留扩展空间

### 2.5 内存管理策略

#### 技术选型：RAII + 智能指针思想

**FFmpeg 资源管理：**

| 资源类型            | 分配函数                              | 释放函数                    | 备注      |
| --------------- | --------------------------------- | ----------------------- | ------- |
| AVFrame         | av\_frame\_alloc                  | av\_frame\_free         | 包含内部缓冲区 |
| AVCodecContext  | avcodec\_alloc\_context3          | avcodec\_free\_context  | 编码器上下文  |
| AVFormatContext | avformat\_alloc\_output\_context2 | avformat\_free\_context | 格式上下文   |
| SwsContext      | sws\_getContext                   | sws\_freeContext        | 颜色转换上下文 |

**关键决策：**

- `frameBuffer` 不单独释放（属于 AVFrame 内部缓冲区）
- 使用 `m_isRecording` 标志防止重复关闭
- 析构函数中统一释放资源

**优势：**

- 避免内存泄漏
- 防止重复释放导致的崩溃
- 异常安全

## 三、关键技术问题与解决方案

### 3.1 播放速度异常问题

**现象：** 录制的视频在 VLC 中播放速度过快

**原因分析：**

1. PTS 计算不准确
2. 编码器时基设置不当
3. 帧率参数未正确传递给编码器

**解决方案：**

```cpp
// 1. 设置编码器帧率
c->framerate = { m_avFormat.frame_rate, 1 };
ost->st->avg_frame_rate = { m_avFormat.frame_rate, 1 };
ost->st->r_frame_rate = { m_avFormat.frame_rate, 1 };

// 2. 基于时间的 PTS 计算
qint64 elapsedMs = currentTime - m_startTime;
video_st.frame->pts = (elapsedMs * m_avFormat.frame_rate) / 1000;
```

### 3.2 程序崩溃问题

**现象：** 点击停止按钮后程序异常终止

**原因：** `frameBuffer` 重复释放（AVFrame 内部已管理）

**解决方案：**

```cpp
// 错误：av_free(ost->frameBuffer);  // 不需要！
// 正确：只释放 frame
av_frame_free(&ost->frame);
ost->frameBuffer = nullptr;
```

### 3.3 视频文件为空问题

**现象：** 录制完成后文件只有 1KB

**原因：** 信号槽未正确连接，视频帧未传递给编码线程

**解决方案：**

```cpp

        m_saveFileThread, SLOT(slot_writeVideoFrame(QImage)));
```

## 四、音视频同步与帧稳帧优化 【2026.4.14 完成】

### 4.1 优化背景

**原有问题：**

1. **音视频不同步**：音频和视频独立编码，没有时间戳对齐机制
2. **帧率不稳定**：采集帧率与编码帧率不匹配，导致播放卡顿或快进
3. **内存风险**：无队列缓冲机制，高负载时可能丢失帧或内存溢出
4. **线程阻塞**：直接在信号槽中编码，可能阻塞数据采集

**优化目标：**

- 实现音视频时间戳同步
- 实现帧稳帧机制（丢帧/补帧）
- 引入线程安全队列缓冲
- 分离采集与编码线程

---

### 4.2 架构优化设计

**优化前架构：**

```
采集线程 ──信号槽──> 编码线程（直接编码）
                          ↓
                      写入文件
```

**优化后架构：**

```
采集线程 ──信号槽──> 入队操作（QMutex保护）
                          ↓
                    ┌─────────────────┐
                    │  线程安全队列     │
                    │  (QList + QMutex)│
                    └────────┬────────┘
                             ↓
                    run() 循环取队
                             ↓
                    ┌─────────────────┐
                    │  音视频同步判断   │
                    │  (av_compare_ts) │
                    └────────┬────────┘
                             ↓
                    ┌─────────────────┐
                    │  帧稳帧处理       │
                    │  (丢帧/补帧)      │
                    └────────┬────────┘
                             ↓
                        编码写入
```

### 4.4 线程安全队列实现

#### 4.4.1 视频帧入队

```cpp
void SaveVideoFileThread::videoDataQueue_Input(uint8_t *buffer, int buffer_size)
{
    QMutexLocker locker(&m_videoMutex);
    
    //优化 队列大小限制，防止内存溢出
    if (m_videoDataList.size() >= MAX_VIDEO_QUEUE_SIZE) {
        BufferDataNode* oldNode = m_videoDataList.takeFirst();
        delete[] oldNode->buffer;
        delete oldNode;
    }
    
    BufferDataNode* node = new BufferDataNode;
    node->buffer = new uint8_t[buffer_size];
    memcpy(node->buffer, buffer, buffer_size);
    node->bufferSize = buffer_size;
    node->time = QDateTime::currentMSecsSinceEpoch();
    
    m_videoDataList.push_back(node);
}
```

#### 4.4.2 视频帧出队

```cpp
BufferDataNode* SaveVideoFileThread::videoDataQueue_get()
{
    QMutexLocker locker(&m_videoMutex);
    
    if (m_videoDataList.isEmpty()) {
        return nullptr;
    }
    
    return m_videoDataList.takeFirst();
}
```

### 4.5 音视频同步机制

#### 4.5.1 同步原理

使用 FFmpeg 的 `av_compare_ts` 函数比较音视频时间戳：

```cpp
//优化 音视频同步比较
int ret = av_compare_ts(video_pts, video_st.enc->time_base,
                        audio_pts, audio_st.enc->time_base);
```

- `ret < 0`：视频落后，应写视频帧
- `ret > 0`：音频落后，应写音频帧
- `ret = 0`：同步，按默认顺序

#### 4.5.2 run() 主循环

```cpp
void SaveVideoFileThread::run()
{
    while (!isStop) {
        //优化 音视频同步判断
        int ret = av_compare_ts(video_pts, video_st.enc->time_base,
                                audio_pts, audio_st.enc->time_base);
        
        if (ret <= 0) {
            // 视频落后或同步，写视频帧
            write_video_frame_from_queue();
        } else {
            // 音频落后，写音频帧
            write_audio_frame_from_queue();
        }
        
        QThread::msleep(1);  // 避免CPU空转
    }
}
```

### 4.6 帧稳帧机制

#### 4.6.1 丢帧策略

当帧时间戳超前于期望时间时，丢弃该帧：

```cpp
//优化 丢帧判断：帧时间戳超前太多则丢弃
if (node->time - m_videoBeginTime > (video_pts + 2) * 40) {
    // 丢弃该帧
    delete[] node->buffer;
    delete node;
    continue;
}
```

#### 4.6.2 补帧策略

当帧时间戳落后于期望时间时，复制上一帧：

```cpp
//优化 补帧判断：帧时间戳落后则复制上一帧
if (lastVideoNode != nullptr && 
    node->time - m_videoBeginTime > (video_pts + 1) * 40) {
    // 复制上一帧进行补帧
    write_video_frame(lastVideoNode);
    video_pts++;
}
```

### 4.7 YUV420P 数据正确复制

**问题：** 原代码将 YUV 数据复制到错误位置，导致录制文件只有 1KB。

**错误代码：**

```cpp
// 错误：复制到 frameBuffer 而非 frame->data
memcpy(video_st.frameBuffer, node->buffer, node->bufferSize);
```

**正确代码：**

```cpp
//优化 正确复制 YUV420P 数据到帧的三个平面
int y_size = video_st.enc->width * video_st.enc->height;
memcpy(video_st.frame->data[0], node->buffer, y_size);           // Y 平面
memcpy(video_st.frame->data[1], node->buffer + y_size, y_size / 4); // U 平面
memcpy(video_st.frame->data[2], node->buffer + y_size + y_size / 4, y_size / 4); // V 平面
```

**YUV420P 内存布局：**

```
┌─────────────────────────────────────┐
│           Y 平面 (width × height)    │
│          亮度分量，全分辨率            │
├──────────────────┬──────────────────┤
│    U 平面         │    V 平面         │
│ (width/2 × height/2)               │
│   色度分量，1/4 分辨率                │
└──────────────────┴──────────────────┘
```
