功能范围与数据库设计

选定架构后，需要确定第一期实现的完整功能清单

数据库表设计（草案）

┌─────────────────────────────────────────────────────┐
│  users (用户表)                                      │
│  id | tel | nickname | password | role | created_at  │
│      role: 0=观众, 1=主播                             │
├─────────────────────────────────────────────────────┤
│  rooms (直播间表)                                     │
│  id | user_id | title | stream_key | status | viewers │
│      status: 0=离线, 1=直播中                          │
│      stream_key: 主播推流时使用的密钥                    │
├─────────────────────────────────────────────────────┤
│  (可选) follows (关注表)                              │
│  follower_id | followed_id                           │
└─────────────────────────────────────────────────────┘

第一期功能范围选择

A. MVP 最小版本（推荐先做）

注册/登录 → 主播创建房间获取推流密钥 → 观众获取直播间列表 → 拉流观看

协议：注册、登录、创建房间、房间列表、获取流密钥
2张表：users + rooms

B. 完整社交版本

MVP + 关注主播 + 弹幕聊天 + 观看人数实时更新 + 直播回放

协议增加：关注、弹幕、在线人数推送
4+张表：users + rooms + follows + chats + replays

Launcher 之后的应用启动方式

A. Launcher 启动子进程

Launcher 登录后，QProcess::start() 启动 Recorder/Player.exe，通过命令行参数传递用户信息和服务器地址

简单直接，子应用无需关心登录逻辑

推荐先做 MVP 版本，把核心链路跑通。请选择功能范围 A 或 B。
