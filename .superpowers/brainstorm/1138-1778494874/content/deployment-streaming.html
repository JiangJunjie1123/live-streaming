部署与推流架构

选定 C++ 技术栈后，需要确定服务器部署位置和 RTMP 推流方案

第一步：服务器部署在哪？

A. Linux 虚拟机 (192.168.136.137)

参考现有 ssh_vm.py，把 C++ epoll 服务器部署到 Linux VM

优点

  • 现有 demo 代码就是 Linux 的，直接编译运行
  • epoll 性能好，生产环境标准
  • 可以顺便部署 nginx-rtmp

缺点

  • 需要交叉编译或用 Linux 编译
  • 调试不如本地方便

B. 本地 Windows

服务器直接跑在当前 Windows 机器上

优点

  • 调试方便，无需SSH
  • 编译环境一致

缺点

  • netapi 用 WinSock2 但 demo 用 epoll，两套代码要整合
  • 不适合生产环境
  • 其他客户端需要知道本机IP

第二步：RTMP 推流怎么走？

方案A推流链路（推荐）
┌──────────────┐    RTMP推流     ┌─────────────────┐    RTMP拉流     ┌──────────────┐
│ VideoRecoder │ ──────────────→ │  nginx-rtmp     │ ──────────────→ │ MediaPlayer  │
│  (主播端)     │   port 1935    │  (Linux VM)     │   port 1935    │  (观众端)     │
└──────┬───────┘                 └────────┬────────┘                 └──────────────┘
       │                                  │
       │ ① 登录/获取推流密钥                │
       │    TCP :8000                     │
       └──────────────────────→ ┌─────────────────┐
                                │  C++ 业务服务器   │
                                │  epoll + MySQL   │
                                │  (Linux VM :8000)│
                                └─────────────────┘

业务服务器只负责：用户认证、房间管理、流密钥分配。音视频数据走 nginx-rtmp，不经过业务服务器。

推荐部署到 Linux VM（方案A），可以同时跑 C++ 业务服务器 + nginx-rtmp 媒体服务器。请选择并告诉我你的想法。
