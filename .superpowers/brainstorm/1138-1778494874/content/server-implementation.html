服务器端实现路线

资料中提供了两套服务器框架，需要选择基础路线

两套框架对比

A. 基于 Epoll Demo 扩展（推荐）

使用 serverDemo（epoll + 线程池 + MySQL），补充完整的注册/登录/房间管理逻辑

TCPKernel (单例)
  ├── Block_Epoll_Net (epoll 网络)
  ├── Thread_Pool (处理数据包)
  ├── CMysql (数据库)
  └── CLogic (业务逻辑→注册/登录/房间)

优点

  • Linux 原生，epoll 高性能
  • 线程池已封装好
  • MySQL 操作类已就绪

缺点

  • 需要补充 CLogic 业务逻辑
  • 协议需要扩展

B. 基于 netapi (Qt) 重新实现

用 netapi 的 TcpServerMediator + Qt 信号槽做服务器，客户端用 TcpClientMediator

INetMediator (Qt信号槽)
  ├── TcpServerMediator (服务器)
  ├── TcpClientMediator (客户端→Launcher)
  └── 自建业务处理 + MySQL

优点

  • 客户端可直接复用 netapi
  • Qt 信号槽开发体验一致

缺点

  • 服务器需要 Qt 运行时(~20MB)
  • epoll 不如原生灵活

客户端对接方案

方案 A：两端不同协议

服务器使用 epoll demo 自定义协议
客户端用 Qt + socket 自行对接

协议格式：4字节长度 + 结构体数据
客户端直接 send/recv 打包结构体

方案 B：两端统一 netapi

服务器和客户端都用 netapi
通过 SIG_ReadyData 信号接收数据

协议格式一样
开发体验统一，但服务器需 Qt 依赖

推荐方案A：服务器用 epoll demo 做高性能后端，客户端用 Qt socket 直接对接二进制协议（格式一致，都是 4 字节头 + 结构体）。
