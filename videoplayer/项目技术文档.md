# 视频播放器项目技术文档

## 项目概述

本项目是一个基于 **Qt + FFmpeg + SDL2** 开发的跨平台视频播放器，支持本地视频文件播放和网络流媒体播放（包括 HLS、RTMP、RTSP 等协议）。项目采用多线程架构，实现了音视频同步、硬件加速渲染、进度条拖拽等完整功能。

## 二、FFmpeg 核心技术原理详解

### 2.1 FFmpeg 架构概述

FFmpeg 是一套完整的音视频解决方案，采用模块化设计，各库之间职责清晰：

```
┌─────────────────────────────────────────────────────────────┐
│                      FFmpeg 架构                            │
├─────────────┬─────────────┬─────────────┬───────────────────┤
│ libavformat │ libavcodec  │ libavutil   │ libswscale        │
│  (格式封装)  │  (编解码)    │  (工具库)   │  (图像转换)        │
├─────────────┼─────────────┼─────────────┼───────────────────┤
│ libswresample│ libavfilter │ libavdevice │ libpostproc       │
│  (音频转换)  │  (滤镜处理)  │  (设备输入)  │  (后处理)         │
└─────────────┴─────────────┴─────────────┴───────────────────┘
```

**各库核心职责**:

| 库名                | 核心职责      | 典型应用场景                         |
| ----------------- | --------- | ------------------------------ |
| **libavformat**   | 封装格式解析与生成 | 打开视频文件、解析MP4/MKV结构、网络流读取       |
| **libavcodec**    | 音视频编解码    | H.264/H.265解码、AAC/MP3编码、编解码器管理 |
| **libavutil**     | 通用工具函数    | 内存管理、数学运算、数据结构、错误处理            |
| **libswscale**    | 图像格式转换    | YUV转RGB、图像缩放、色彩空间转换            |
| **libswresample** | 音频重采样     | 采样率转换、声道布局转换、采样格式转换            |
| **libavfilter**   | 音视频滤镜     | 水印添加、裁剪、旋转、音频均衡器               |
| **libavdevice**   | 设备输入输出    | 摄像头捕获、屏幕录制、DirectShow接口        |

**设计哲学**：FFmpeg 采用分层架构，上层库依赖下层库，但下层库保持独立。这种设计使得开发者可以根据需求选择使用哪些模块，而不必引入整个框架。

### 2.2 音视频编解码原理

#### 2.2.1 为什么需要编解码

**原始视频数据量巨大**:

- 1920×1080 @ 30fps 的原始视频，每秒数据量约为 1920 × 1080 × 3字节 × 30帧 ≈ **178MB/s**
- 一部2小时的电影需要约 **1.28TB** 存储空间
- 显然无法直接存储和传输

**编码的核心目标**:

1. **去除冗余信息**: 利用视频的时间和空间相关性
2. **视觉无损压缩**: 去除人眼不敏感的信息
3. **可控压缩比**: 在质量和文件大小之间权衡

#### 2.2.2 视频压缩的核心原理

**空间冗余消除**：
同一帧图像中，相邻像素通常具有相似的颜色值。例如，天空区域可能包含数百万个相似的蓝色像素。编码器通过预测和变换，将这些相似像素用更少的参数表示。

**时间冗余消除**：
相邻帧之间变化很小（通常只有5-10%的像素变化）。编码器只记录帧之间的差异，而不是完整图像。

```
帧1: [完整图像] ──→ 帧2: [只记录变化部分] ──→ 帧3: [只记录变化部分]
        ↑                    ↑                      ↑
    I帧(关键帧)          P帧(预测帧)              P帧(预测帧)
    独立解码             依赖前一帧              依赖前一帧
```

#### 2.2.3 视频编码流程深度解析

**第一步：帧类型判断与GOP结构**

视频编码使用三种帧类型，形成 GOP（Group of Pictures）结构：

| 帧类型    | 名称    | 特点          | 压缩比      | 典型大小(1080p) |
| ------ | ----- | ----------- | -------- | ----------- |
| **I帧** | 关键帧   | 完整图像，独立解码   | 低（约1:7）  | 50-100KB    |
| **P帧** | 前向预测帧 | 参考前一帧，记录差异  | 中（约1:20） | 10-20KB     |
| **B帧** | 双向预测帧 | 参考前后帧，压缩率最高 | 高（约1:50） | 5-10KB      |

GOP 结构示例（IPBBPBBI...）:

```
时间轴:  t0    t1    t2    t3    t4    t5    t6    t7
帧类型:  I  ←  P  ←  B  ←  B  ←  P  ←  B  ←  B  ←  I
         ↑     ↑           ↑     ↑           ↑     ↑
      独立   参考I帧    参考I/P帧  参考P帧   参考P帧  独立
```

**GOP大小的设计考量**：

- **短GOP（1-2秒）**：Seek响应快，但压缩率低，I帧频繁导致码率波动
- **长GOP（4-10秒）**：压缩率高，码率稳定，但Seek时需要解码更多帧
- **典型值**：24-30帧（1秒@24fps，2秒@30fps）

**第二步：运动估计与补偿（Motion Estimation & Compensation）**

核心思想：物体在相邻帧之间通常是移动的，而不是变化的。

**运动估计过程**：

1. 将当前帧划分为宏块（通常是16×16像素）
2. 在参考帧中搜索最佳匹配块
3. 记录运动向量（水平和垂直位移）
4. 计算残差（实际像素与预测像素的差值）

**搜索算法**：

- **全搜索**：遍历所有可能位置，精度最高但计算量巨大
- **三步搜索**：快速收敛的启发式算法
- **钻石搜索**：适用于运动较小的场景
- **六边形搜索**：平衡精度和速度

**第三步：DCT 变换（离散余弦变换）**

将图像从空间域转换到频率域：

- **空间域**：每个像素的颜色值
- **频率域**：图像中不同频率的成分

**为什么转换？**

- 自然图像的能量主要集中在低频区域
- 高频信息（细节）对人眼不敏感，可以大量丢弃

**DCT数学原理**：

```
8×8块的二维DCT变换:

F(u,v) = (1/4)C(u)C(v)ΣΣf(x,y)cos[(2x+1)uπ/16]cos[(2y+1)vπ/16]

其中:
- f(x,y): 空间域像素值
- F(u,v): 频率域系数
- C(u),C(v): 归一化系数（u,v=0时为1/√2，否则为1）
```

**频率系数分布**：

```
空间域图像块 (8×8)          频率域系数 (8×8)
┌──┬──┬──┬──┬──┬──┬──┬──┐   ┌────┬────┬────┬────┬───┬───┬───┬───┐
│10│12│11│13│12│11│10│12│   │120 │ 45 │ 20 │ 8  │ 3 │ 2 │ 1 │ 0 │ ← 直流分量+低频
│11│13│12│14│13│12│11│13│   │ 40 │ 18 │ 9  │ 4  │ 2 │ 1 │ 0 │ 0 │
│12│14│13│15│14│13│12│14│   │ 15 │ 8  │ 4  │ 2  │ 1 │ 0 │ 0 │ 0 │
│... (平滑渐变) ...      │   │ 6  │ 3  │ 2  │ 1  │ 0 │ 0 │ 0 │ 0 │
│                        │   │ 2  │ 1  │ 0  │ 0  │ 0 │ 0 │ 0 │ 0 │ ← 高频分量
│                        │   │ 1  │ 0  │ 0  │ 0  │ 0 │ 0 │ 0 │ 0 │   (可大量丢弃)
│                        │   │ 0  │ 0  │ 0  │ 0  │ 0 │ 0  │ 0 │ 0 │
└──┴──┴──┴──┴──┴──┴──┴──┘   └────┴────┴────┴────┴───┴───┴───┴───┘
```

**第四步：量化处理（Quantization）**

量化是**有损压缩**的核心步骤，通过降低精度来减少数据量：

**量化原理**：

```
原始DCT系数:    [120, 45, 20, 8, 3, 2, 1, 0, ...]
量化步长:       [  1,  2,  4, 8, 8, 8, 8, 8, ...]

量化后:         [120, 22,  5, 1, 0, 0, 0, 0, ...]
                 ↑    ↑   ↑  ↑  ↑
               保留  保留 保留 粗略 丢弃（变为0）
```

**量化表设计原理**：

- 低频分量使用小步长（精细量化）
- 高频分量使用大步长（粗糙量化或直接丢弃）
- 人眼对高频信息不敏感，丢弃后视觉影响小

**量化参数（QP）**：

- QP值越小，量化步长越小，质量越高，文件越大
- QP值越大，量化步长越大，质量越低，文件越小
- H.264中QP范围通常为0-51

**第五步：熵编码（Entropy Coding）**

利用数据的统计特性进行无损压缩：

**Huffman 编码原理**：
基于符号出现频率构建最优前缀码，高频符号用短码，低频符号用长码。

**构建过程**：

1. 统计各符号出现频率
2. 构建Huffman树（贪心算法，每次合并频率最小的两个节点）
3. 从根到叶子的路径即为编码

**编码效率示例**：

```
原始数据: [A, A, A, B, B, C, D]  (7个符号，共56位)
频率统计: A(3), B(2), C(1), D(1)

编码结果:
A: 0      (1位，频率最高，编码最短)
B: 10     (2位)
C: 110    (3位)
D: 111    (3位)

压缩后: 3×1 + 2×2 + 1×3 + 1×3 = 13位
压缩比: 56/13 ≈ 4.3:1
```

**CABAC（上下文自适应二进制算术编码）**：
H.264/H.265使用的高级熵编码方法，比Huffman编码效率更高（通常提升10-20%），但计算复杂度也更高。

#### 2.2.4 解码流程

解码是编码的逆过程：

```
压缩数据
    │
    ▼
┌───────────────┐
│   熵解码       │  ← 还原DCT系数（无损）
└───────┬───────┘
        │
        ▼
┌───────────────┐
│   反量化       │  ← 恢复系数精度（有损失，不可逆）
└───────┬───────┘
        │
        ▼
┌───────────────┐
│   反DCT变换    │  ← 从频率域转回空间域
└───────┬───────┘
        │
        ▼
┌───────────────┐
│   运动补偿     │  ← 根据运动向量还原图像
└───────┬───────┘
        │
        ▼
还原的视频帧 (YUV格式)
```

#### 2.2.5 FFmpeg 编解码核心概念

**核心数据结构关系**：

FFmpeg使用面向对象的设计思想，通过上下文结构体管理编解码状态：

- **AVFormatContext**：管理整个媒体文件，包含流信息、时长、元数据
- **AVStream**：表示一个音视频流，包含时间基、编码参数
- **AVCodecContext**：编解码器上下文，管理编解码状态
- **AVCodec**：编解码器本身，包含编解码函数指针
- **AVPacket**：压缩后的数据包，从文件读取或写入文件
- **AVFrame**：解码后的原始帧，包含图像或音频数据

**编解码流程设计**：

FFmpeg 采用 **send/receive** 分离式设计，支持异步处理：

```
传统方式（同步）:           FFmpeg方式（异步）:
┌─────────────┐            ┌─────────────┐
│ 输入数据包   │            │ 输入数据包   │
└──────┬──────┘            └──────┬──────┘
       │                          │
       ▼                          ▼ send_packet()
┌─────────────┐            ┌─────────────┐
│  解码/编码   │            │  内部缓冲区  │
│  （阻塞）    │            │  （异步）    │
└──────┬──────┘            └──────┬──────┘
       │                          │
       ▼                          ▼ receive_frame()
┌─────────────┐            ┌─────────────┐
│  输出帧     │            │  输出帧     │
└─────────────┘            └─────────────┘

优势：
1. 可以批量输入数据，提高吞吐量
2. 编解码器内部可以优化缓冲和并行处理
3. 更适合硬件加速（GPU编解码）
```

### 2.3 格式解析原理

#### 2.3.1 封装格式的本质

封装格式（Container Format）是**组织**音视频数据的**文件结构标准**，它解决以下问题：

1. **多路复用**: 将视频流、音频流、字幕流打包在一起
2. **同步信息**: 记录各流的时间戳，确保播放时同步
3. **元数据**: 存储标题、作者、时长等信息
4. **索引信息**: 支持快速定位和Seek

**封装 vs 编码的区别**：

| 概念       | 作用       | 类比      |
| -------- | -------- | ------- |
| **编码格式** | 压缩音视频数据  | ZIP压缩算法 |
| **封装格式** | 组织压缩后的数据 | ZIP文件结构 |

#### 2.3.2 常见封装格式对比

| 格式       | 开发方       | 特点                | 适用场景        |
| -------- | --------- | ----------------- | ----------- |
| **MP4**  | MPEG      | 通用性强，网络友好，支持流媒体   | 互联网视频、移动设备  |
| **MKV**  | 开源社区      | 开源免费，功能丰富，支持多音轨字幕 | 高清收藏、动漫字幕组  |
| **FLV**  | Adobe     | 文件头小，适合网络渐进下载     | 早期网络视频、直播推流 |
| **AVI**  | Microsoft | 历史悠久，兼容性好，但功能简单   | 老视频、简单编辑    |
| **TS**   | MPEG      | 容错性强，适合传输误码环境     | 数字电视、IPTV直播 |
| **M3U8** | Apple     | 播放列表格式，支持自适应码率    | HLS直播/点播    |

#### 2.3.3 解封装的核心流程

**第一步：探测格式**

FFmpeg 通过读取文件头部数据，识别封装格式：

**文件签名（Magic Number）**：

```
MP4/MOV:  00 00 00 XX 66 74 79 70  (ftyp 标记)
MKV:      1A 45 DF A3               (EBML 标记)
FLV:      46 4C 56 01               ("FLV" + 版本)
AVI:      52 49 46 46 XX XX XX XX 41 56 49 20  (RIFF...AVI )
TS:       47                        (同步字节0x47，每188字节)
```

**探测机制**：
FFmpeg维护一个格式探测分数表，读取文件头部后，各格式解析器尝试解析，返回匹配分数，最高分者被选中。

**第二步：解析流信息**

读取文件的元数据段，获取音视频流参数：

**MP4文件的moov box结构**：

```
┌─────────────────────────────────────┐
│ moov (Movie Header)                 │
│ ├── mvhd (Movie Header)             │
│ │   ├── 时长、时间尺度、创建时间      │
│ │   └── 默认音量、速率               │
│ │                                   │
│ ├── trak[0] (Video Track)           │
│ │   ├── tkhd (Track Header)         │
│ │   │   └── 轨道ID、时长、宽高        │
│ │   ├── mdia (Media)                │
│ │   │   ├── mdhd (Media Header)     │
│ │   │   │   └── 时间尺度、语言        │
│ │   │   ├── hdlr (Handler)          │
│ │   │   │   └── 类型: vide          │
│ │   │   └── minf (Media Info)       │
│ │   │       └── stbl (Sample Table) │
│ │   │           ├── stsd (Sample Desc) ← 编码格式: avc1 (H.264)
│ │   │           ├── stts (Time to Sample) ← 帧时间戳映射
│ │   │           ├── stsc (Sample to Chunk) ← 数据块映射
│ │   │           ├── stsz (Sample Size) ← 每帧大小
│ │   │           └── stco (Chunk Offset) ← 数据偏移位置
│ │   └── ...                         │
│ │                                   │
│ └── trak[1] (Audio Track)           │
│     └── ... (类似结构，类型: soun)   │
└─────────────────────────────────────┘
```

**关键box说明**：

- **stts**：时间戳映射表，将sample序号映射到解码时间（DTS）
- **stss**：同步sample表，标记关键帧位置，用于Seek
- **ctts**：合成时间偏移表，处理B帧的PTS/DTS差异

**第三步：读取数据包**

根据索引信息，按时间顺序读取压缩数据：

**时间轴上的数据分布**：

```
时间:    0s      1s      2s      3s      4s      5s
         │       │       │       │       │       │
视频:    [I][P][P][P][P][P][P][P][P][P][I][P][P][P]...
         ↑                   ↑               ↑
       关键帧              可Seek点        下一个GOP

音频:    [AAC][AAC][AAC][AAC][AAC][AAC][AAC][AAC]...
         每帧1024采样

封装文件中的物理存储:
┌────────┬────────┬────────┬────────┬────────┬────────┐
│ 视频帧 │ 音频帧 │ 视频帧 │ 音频帧 │ 视频帧 │ 音频帧 │
│  #1    │  #1    │  #2    │  #2    │  #3    │  #3    │
│ (I帧)  │ (AAC)  │ (P帧)  │ (AAC)  │ (P帧)  │ (AAC)  │
└────────┴────────┴────────┴────────┴────────┴────────┘
```

**交错存储的优势**：

- 播放时顺序读取，减少磁盘寻道
- 网络流式传输友好
- 支持渐进式下载播放

#### 2.3.4 FFmpeg 解封装设计

**AVFormatContext 的核心作用**：

AVFormatContext 是 FFmpeg 中管理媒体文件的**核心上下文**，它：

1. **管理协议**: 处理文件、网络、设备等不同的输入源
2. **管理格式**: 自动识别或指定封装格式
3. **管理流**: 维护音视频流数组，处理多路流
4. **管理时间**: 统一不同流的时间基，提供Seek功能

**流（AVStream）的概念**：

一个媒体文件可以包含多个流，每个流是独立的：

```
典型电影文件结构:
┌─────────────────────────────────────────┐
│           电影.mp4                      │
│  ┌─────────────────────────────────┐    │
│  │  Stream #0: Video (H.264)       │    │
│  │  - 分辨率: 1920×1080            │    │
│  │  - 帧率: 24fps                  │    │
│  │  - 码率: 8Mbps                  │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  Stream #1: Audio (AAC)         │    │
│  │  - 采样率: 48000Hz              │    │
```

### 2.4 网络流处理原理

#### 2.4.1 流媒体 vs 本地文件

**本地文件播放特点**：

- 文件完整存储在本地磁盘
- 可以任意Seek到任意位置
- 读取速度稳定，不受网络影响
- 文件头信息完整可用

**流媒体播放特点**：

- 数据实时从网络获取
- Seek受服务器支持程度限制
- 网络波动影响播放流畅度
- 可能需要边下载边解析

#### 2.4.2 网络协议对比

| 协议         | 传输层      | 特点                      | 延迟        | 适用场景         |
| ---------- | -------- | ----------------------- | --------- | ------------ |
| **HTTP**   | TCP      | 通用性强，穿透性好，缓存友好          | 中         | 点播、文件下载      |
| **HLS**    | HTTP/TCP | 基于HTTP，自适应码率，切片传输       | 高(10-30s) | 直播、点播（iOS首选） |
| **DASH**   | HTTP/TCP | 标准化MPEG-DASH，多厂商支持      | 高(10-30s) | 跨平台点播        |
| **RTMP**   | TCP      | Adobe私有协议，低延迟，Flash时代主流 | 低(2-5s)   | 直播推流（逐渐淘汰）   |
| **RTSP**   | TCP/UDP  | 支持双向控制，标准协议             | 低(1-3s)   | 监控、视频会议      |
| **WebRTC** | UDP      | 浏览器原生支持，超低延迟            | 极低(<1s)   | 视频会议、连麦直播    |
| **SRT**    | UDP      | 开源，抗丢包，低延迟              | 低(1-3s)   | 专业直播、远距离传输   |
| **RIST**   | UDP      | 开源标准，广播级可靠性             | 低         | 广电行业         |

#### 2.4.3 HLS 协议深度解析

**HLS 设计哲学**：

HLS (HTTP Live Streaming) 的核心思想是**化整为零**：

```
传统流媒体:                    HLS:
┌──────────────────┐          ┌──────────────────┐
│  连续数据流       │          │  切片1  切片2  切片3 │
│  ██████████████  │    →     │  ████   ████   ████ │
│  一旦断开就失败   │          │  某个切片失败可重试  │
└──────────────────┘          └──────────────────┘

优势:
1. 基于HTTP，兼容所有CDN和缓存服务器
2. 切片独立，某个切片失败不影响整体
3. 天然支持自适应码率（不同码率对应不同切片列表）
4. 实现简单，服务器只需静态文件服务
```

**HLS 工作流程**：

```
服务器端处理流程:

视频源 ──→ 编码器 ──→ 切片器 ──→ 文件服务器
            │           │            │
            │           │            ├── index.m3u8 (播放列表)
            │           │            ├── 360p_001.ts
            │           │            ├── 360p_002.ts
            │           │            ├── 720p_001.ts
            │           │            └── 720p_002.ts
            │           │
            │           └── 每N秒切一个片段
            │               更新M3U8列表
            │
            └── 同时输出多码率
                360p / 720p / 1080p

客户端播放流程:

1. 下载 M3U8 播放列表
   ┌─────────────────────────────────────┐
   │ #EXTM3U                             │
   │ #EXT-X-VERSION:3                    │
   │ #EXT-X-TARGETDURATION:10            │
   │ #EXT-X-MEDIA-SEQUENCE:0             │
   │                                     │
   │ #EXTINF:10.0,                       │
   │ output0.ts                          │
   │ #EXTINF:10.0,                       │
   │ output1.ts                          │
   │ #EXTINF:10.0,                       │
   │ output2.ts                          │
   │ #EXT-X-ENDLIST                      │
   └─────────────────────────────────────┘

2. 按顺序下载TS切片并播放

3. 直播场景下定期重新下载M3U8获取新切片
```

**M3U8 标签说明**：

- **#EXTM3U**: 文件头，标识M3U格式
- **#EXT-X-VERSION**: HLS协议版本
- **#EXT-X-TARGETDURATION**: 最大切片时长
- **#EXT-X-MEDIA-SEQUENCE**: 第一个切片的序号（直播用）
- **#EXTINF**: 切片时长
- **#EXT-X-ENDLIST**: 点播结束标记（直播没有此标签）
- **#EXT-X-STREAM-INF**: 多码率列表（自适应码率）

#### 2.4.4 网络配置与优化

FFmpeg 提供丰富的网络配置选项，用于处理各种网络场景：

**超时与缓冲配置**：

- **stimeout**: 套接字超时时间，防止无限等待
- **timeout**: 连接超时时间
- **max_delay**: 最大延迟限制，控制缓冲大小
- **buffer_size**: 接收缓冲区大小，影响网络吞吐量

**HLS 专用配置**：

- **live_start_index**: 直播流起始播放位置，0表示从第一个切片开始
- **hls_allow_cache**: 是否允许缓存切片
- **hls_playlist_load_attempts**: 播放列表加载重试次数
- **seekable**: 强制启用Seek支持，某些流需要显式设置

**RTSP 配置**：

- **rtsp_transport**: 传输协议选择（tcp/udp），TCP更稳定
- **rtsp_flags**: RTSP特定标志

### 2.5 像素格式与图像转换

#### 2.5.1 为什么需要像素格式转换

**不同系统使用不同的像素格式**：

- **视频编码**: 通常使用YUV格式（YUV420P、NV12等）
- **显示器**: 通常使用RGB格式（RGB24、RGB32等）
- **GPU处理**: 可能使用特殊的压缩格式

**YUV vs RGB**：

| 特性   | RGB             | YUV                  |
| ---- | --------------- | -------------------- |
| 存储方式 | 每个像素包含R、G、B三个分量 | 亮度Y和色度UV分离存储         |
| 数据量  | 较大（每个像素3-4字节）   | 较小（YUV420P为1.5字节/像素） |
| 适用场景 | 显示、图像处理         | 视频编码、传输              |
| 人眼感知 | 直接对应颜色          | 利用人眼对亮度更敏感的特性        |

#### 2.5.2 YUV 色彩空间详解

YUV 将颜色信息分解为：

- **Y (Luma)**: 亮度信息，人眼最敏感
- **U (Cb)**: 蓝色色度差
- **V (Cr)**: 红色色度差

**RGB 与 YUV 转换公式**：

```
RGB 转 YUV:
Y = 0.299R + 0.587G + 0.114B
U = -0.169R - 0.331G + 0.5B + 128
V = 0.5R - 0.419G - 0.081B + 128

YUV 转 RGB:
R = Y + 1.402(V - 128)
G = Y - 0.344(U - 128) - 0.714(V - 128)
B = Y + 1.772(U - 128)
```

**YUV420P 内存布局**：

```
YUV420P 采用 4:2:0 采样，色度分量分辨率减半:

┌────────────────────────────────────────┐
│ Y 平面 (全分辨率)                       │
│ 每个像素1字节，共 width × height 字节   │
│                                        │
│ Y00 Y01 Y02 Y03 ...                    │
│ Y10 Y11 Y12 Y13 ...                    │
│ ...                                    │
├────────────────────────────────────────┤
│ U 平面 (1/4分辨率)                      │
│ 每4个Y共享1个U，共 (width/2) × (height/2) │
│                                        │
│ U00 U01 ...                            │
│ U10 U11 ...                            │
├────────────────────────────────────────┤
│ V 平面 (1/4分辨率)                      │
│ 每4个Y共享1个V，共 (width/2) × (height/2) │
│                                        │
│ V00 V01 ...                            │
│ V10 V11 ...                            │
└────────────────────────────────────────┘

总数据量: width × height × 1.5 字节
压缩比: 相比RGB24节省50%空间
```

**YUV采样格式对比**：

| 格式     | Y采样 | UV采样     | 数据量      | 应用场景     |
| ------ | --- | -------- | -------- | -------- |
| YUV444 | 1:1 | 1:1      | 3字节/像素   | 专业视频处理   |
| YUV422 | 1:1 | 2:1水平    | 2字节/像素   | 电视广播     |
| YUV420 | 1:1 | 2:1水平和垂直 | 1.5字节/像素 | 视频编码、互联网 |
| YUV411 | 1:1 | 4:1水平    | 1.5字节/像素 | 早期数字视频   |

#### 2.5.3 图像缩放与转换

**SwsScale 工作原理**：

SwsScale 是 FFmpeg 的图像转换库，支持：

- 像素格式转换（YUV→RGB、RGB→YUV等）
- 图像缩放（放大/缩小）
- 色彩空间转换（BT.601、BT.709等）

**缩放算法选择**：

| 算法                    | 速度  | 质量  | 适用场景     |
| --------------------- | --- | --- | -------- |
| **SWS_FAST_BILINEAR** | 最快  | 一般  | 实时预览、缩略图 |
| **SWS_BILINEAR**      | 快   | 较好  | 一般用途     |
| **SWS_BICUBIC**       | 中等  | 好   | 高质量缩放    |
| **SWS_LANCZOS**       | 慢   | 最好  | 高质量输出    |
| **SWS_SPLINE**        | 较慢  | 很好  | 专业图像处理   |

**缩放算法原理简述**：

- **最近邻**：取最近的像素值，速度快但锯齿明显
- **双线性**：使用周围4个像素加权平均，平滑但可能模糊
- **双三次**：使用周围16个像素，更平滑的边缘
- **Lanczos**：使用sinc函数，保留更多细节

### 2.6 时间基与同步

#### 2.6.1 时间基的概念

**为什么需要时间基？**

不同编码格式使用不同的时间精度：

- MPEG-2 TS: 90kHz (1/90000秒)
- MP4: 根据文件不同，常见1/1000秒或1/30000秒
- FLV: 1/1000秒

**时间基（Time Base）定义**：

时间基是一个分数，表示时间戳的单位：

```
时间基 = 分子 / 分母

例如:
- {1, 1000} 表示 1/1000 秒 = 1毫秒
- {1, 90000} 表示 1/90000 秒 ≈ 11.11微秒
- {1, 25} 表示 1/25 秒 = 40毫秒（25fps的帧间隔）
```

**时间戳转换**：

```
时间(秒) = 时间戳 × 时间基
         = 时间戳 × (num / den)

示例:
时间戳 = 45000
时间基 = {1, 90000}
时间 = 45000 × (1/90000) = 0.5秒
```

#### 2.6.2 FFmpeg 中的时间基层级

FFmpeg 中有三个层级的时间基：

**1. AVFormatContext 级别（全局时间基）**：

```
时间基: {1, AV_TIME_BASE} = {1, 1000000} (微秒)
用途: 统一所有流的时间戳
字段: duration（总时长）、start_time（开始时间）
```

**2. AVStream 级别（流级别时间基）**：

```
时间基: 根据封装格式不同而不同
用途: 该流内部的时间戳单位
字段: time_base、duration、start_time

常见值:
- MP4视频: {1, 30000} 或 {1, 60000}
- MP4音频: {1, 采样率}
- MPEG-TS: {1, 90000}
```

**3. AVCodecContext 级别（编解码器级别）**：

```
时间基: 用于编解码器的时间戳
用途: 编解码过程中的时间戳转换
字段: time_base
```

#### 2.6.3 时间戳转换方法

**av_rescale_q() 函数**：

用于在不同时基之间转换时间戳：

```
目标时间戳 = av_rescale_q(源时间戳, 源时间基, 目标时间基)

数学原理:
目标时间戳 = 源时间戳 × (源num/源den) / (目标num/目标den)
           = 源时间戳 × 源num × 目标den / (源den × 目标num)

示例:
将MP4流时间戳转换为全局微秒时间:
流时间基 = {1, 30000}
全局时间基 = {1, 1000000}

时间戳 1500（流时间基）:
= 1500 × (1/30000) 秒
= 0.05 秒
= 50000 微秒
```

### 3.2 音频播放原理

#### 3.2.1 数字音频基础

**音频信号数字化过程**：

模拟声音波形通过采样和量化转换为数字信号：

```
模拟声音波形
     ╭─╮    ╭──╮      ╭──╮
────╯  ╰────╯  ╰──────╯  ╰────
     │采样│采样│采样│采样│采样│
     ▼    ▼    ▼    ▼    ▼
    ┌─┐  ┌─┐  ┌─┐  ┌─┐  ┌─┐
    │ │  │ │  │ │  │ │  │ │  ← 量化（转换为数字值）
    └─┘  └─┘  └─┘  └─┘  └─┘
     ↑    ↑    ↑    ↑    ↑
   时间轴上的离散采样点

关键参数:
- 采样率: 每秒采样次数（Hz）
- 采样位数: 每个采样的精度（8/16/24/32位）
- 声道数: 单声道/立体声/5.1环绕
```

**奈奎斯特定理**：
为了准确还原模拟信号，采样率必须至少是信号最高频率的2倍。人耳可听范围约20Hz-20kHz，因此CD音质采用44.1kHz采样率。

**常见音频参数**：

| 参数   | CD音质      | 高清音频            | 语音通话          |
| ---- | --------- | --------------- | ------------- |
| 采样率  | 44100 Hz  | 48000/96000 Hz  | 8000/16000 Hz |
| 采样格式 | 16位有符号    | 24位/32位浮点       | 16位有符号        |
| 声道数  | 2（立体声）    | 2-8             | 1（单声道）        |
| 数据率  | 1411 kbps | 2304-12288 kbps | 64-256 kbps   |

#### 3.2.2 音频播放的工作模式

**拉取模式（Pull Mode）- SDL采用的方式**：

```
应用程序                    SDL音频子系统                    声卡
    │                            │                            │
    │  1. 注册回调函数            │                            │
    │  2. 指定音频格式            │                            │
    │ ─────────────────────────> │                            │
    │                            │  3. 初始化音频设备            │
    │                            │ ─────────────────────────> │
    │                            │                            │
    │                            │  4. 音频设备需要数据（中断）   │
    │ <───────────────────────── │                            │
    │                            │                            │
    │  5. 回调函数被调用          │                            │
    │  - 解码音频数据             │                            │
    │  - 填充到SDL提供的缓冲区     │                            │
    │ ─────────────────────────> │                            │
    │                            │  6. 将数据送入声卡            │
    │                            │ ─────────────────────────> │
    │                            │                            │
    │                            │  7. 播放完成，再次请求数据     │
    │ <───────────────────────── │                            │
    │                            │         （循环）             │

特点:
- 由音频设备驱动数据流
- 实时性要求高，回调中不能阻塞
- 适合低延迟音频播放
```

**推送模式（Push Mode）**：

```
应用程序                    音频系统
    │                            │
    │  1. 主动写入音频数据        │
    │ ─────────────────────────> │
    │                            │
    │  2. 等待播放完成/缓冲空闲   │
    │ <───────────────────────── │
    │                            │
    │  3. 再次写入数据            │
    │ ─────────────────────────> │

特点:
- 由应用程序控制数据流
- 可以实现更复杂的缓冲策略
- 延迟相对较高
```

**两种模式对比**：

| 特性    | 拉取模式    | 推送模式    |
| ----- | ------- | ------- |
| 控制方   | 音频设备    | 应用程序    |
| 延迟    | 低       | 较高      |
| 实现复杂度 | 较高      | 较低      |
| 适用场景  | 实时播放、游戏 | 文件播放、录音 |

#### 3.2.3 SDL 音频回调机制

**回调函数的工作流程**：

```
audio_callback(userdata, stream, len)
         │                    │       │
         │                    │       └── 需要填充的字节数
         │                    │           （由SDL根据缓冲区大小计算）
         │                    │
         │                    └── SDL提供的音频缓冲区
         │                        （需要将解码后的音频数据写入这里）
         │
         └── 用户自定义数据指针
             （通常传递播放器状态结构体）

回调函数内部逻辑:
┌─────────────────────────────────────┐
│ 1. 清零stream缓冲区（避免噪音）      │
├─────────────────────────────────────┤
│ 2. 循环直到填满len字节              │
│    while (len > 0) {                │
│      a. 检查本地音频缓冲区           │
│      b. 如果空了，解码更多数据       │
│      c. 复制数据到stream             │
│      d. 更新指针和计数器             │
│    }                                │
├─────────────────────────────────────┤
│ 3. 使用SDL_MixAudioFormat混音        │
│    （支持音量控制）                  │
└─────────────────────────────────────┘

重要约束:
- 回调在单独的音频线程中执行
- 不能阻塞（不能调用耗时操作）
- 不能访问UI（需要使用线程安全的方式通信）
- 必须快速返回，否则会产生音频卡顿
```

**音频缓冲策略**：

```
音频数据流:

解码器输出 ──→ 音频队列 ──→ 音频缓冲区 ──→ SDL回调 ──→ 声卡
                (PacketQueue)   (环形缓冲)    (实时)

缓冲层次:
1. 解码层: 从文件/网络读取并解码音频包
2. 队列层: PacketQueue存储解码后的音频帧
3. 缓冲层: 音频回调中的本地缓冲区
4. 设备层: SDL内部的音频设备缓冲区

缓冲大小选择:
- 太小: 容易欠载（underrun），产生卡顿
- 太大: 延迟增加，Seek响应慢
- 推荐: 1024-4096采样（约20-80ms延迟）
```

**欠载与过载处理**：

- **欠载（Underrun）**：回调无法及时提供数据，产生静音或噪音。解决方法是增加缓冲区大小或优化解码性能。
- **过载（Overrun）**：数据产生速度超过播放速度，导致延迟累积。解决方法是丢帧或调整播放速度。

#### 3.2.4 音频重采样

**为什么需要重采样**：

```
源音频          目标设备
44100 Hz  ──→  48000 Hz
                不匹配！

或者:
立体声  ──→  单声道
                需要转换！

或者:
32位浮点  ──→  16位整数
                格式不同！
```

**重采样的质量与性能权衡**：

| 算法     | 质量  | CPU占用 | 适用场景    |
| ------ | --- | ----- | ------- |
| 最近邻    | 低   | 极低    | 实时性要求极高 |
| 线性插值   | 中   | 低     | 一般用途    |
| 多相滤波   | 高   | 中     | 高质量要求   |
| Sinc插值 | 最高  | 高     | 专业音频处理  |

**重采样算法原理**：

- **最近邻**：直接取最近的采样点，速度快但可能产生混叠
- **线性插值**：在两个采样点之间线性插值，简单平滑
- **多相滤波**：使用FIR滤波器，更好的抗混叠
- **Sinc插值**：理论上完美的重构，但计算量大

### 3.3 线程管理原理

#### 3.3.1 为什么需要多线程

**视频播放的并发需求**：

```
单线程的问题:
┌─────────────────────────────────────────────────────────────┐
│ 读取文件 → 解码视频 → 解码音频 → 显示图像 → 播放音频 → ... │
└─────────────────────────────────────────────────────────────┘
- 任何一个环节阻塞，整个播放都会卡顿
- 无法利用多核CPU
- 网络读取会阻塞解码

多线程方案:
┌─────────────┐
│  读取线程    │ ──→ 从文件/网络读取数据包
└──────┬──────┘
       │ 放入队列
       ▼
┌─────────────┐     ┌─────────────┐
│  视频队列    │────→│  视频解码线程│ ──→ 解码 → 显示
└─────────────┘     └─────────────┘

┌─────────────┐     ┌─────────────┐
│  音频队列    │────→│  音频播放    │ ──→ SDL回调播放
└─────────────┘     └─────────────┘

优势:
- 读取、解码、播放并行进行
- 网络延迟不影响解码和播放
- 充分利用多核CPU
```

#### 3.3.2 线程同步机制

**互斥锁（Mutex）原理**：

```
互斥锁保护共享资源的访问:

线程A                          线程B
  │                              │
  │  SDL_LockMutex(mutex)        │
  │  ─────────────────────────>  │
  │  获得锁，进入临界区           │
  │                              │
  │  访问共享资源                 │
  │  (修改PacketQueue)           │
  │                              │
  │                              │  SDL_LockMutex(mutex)
  │                              │  ─────────────────────────> 
  │                              │  阻塞等待...
  │                              │
  │  SDL_UnlockMutex(mutex)      │
  │  ─────────────────────────>  │
  │  释放锁                       │  获得锁，继续执行
  │                              │

关键特性:
- 同一时间只有一个线程可以持有锁
- 其他线程尝试加锁会阻塞等待
- 必须成对使用Lock/Unlock
- 防止死锁（避免嵌套锁、确保释放）
```

**死锁预防**：

1. **锁顺序**：所有线程按相同顺序获取多个锁
2. **锁超时**：使用try_lock，超时后放弃
3. **最小持有**：尽量减少持锁时间
4. **避免回调**：持锁时避免调用可能获取其他锁的函数

**条件变量（Condition Variable）原理**：

```
条件变量用于线程间的事件通知:

生产者线程（解码器）          消费者线程（播放器）
        │                              │
        │  解码完成，数据放入队列       │
        │                              │
        │  SDL_LockMutex(mutex)        │
        │  修改队列状态                 │
        │                              │
        │  SDL_CondSignal(cond)   ───> │  唤醒等待的消费者
        │  （通知消费者）               │
        │                              │
        │  SDL_UnlockMutex(mutex)      │
        │                              │
        │                         ┌────┴────┐
        │                         │ 被唤醒， │
        │                         │ 检查条件 │
        │                         │ 满足，继续│
        │                         └────┬────┘
        │                              │

消费者等待过程:
SDL_LockMutex(mutex);
while (队列空) {                    // 必须用while，防止虚假唤醒
    SDL_CondWait(cond, mutex);      // 自动解锁mutex，进入等待
                                    // 被唤醒后自动重新加锁
}
从队列取数据;
SDL_UnlockMutex(mutex);

为什么用while而不是if:
- 防止虚假唤醒（spurious wakeup）
- 多个消费者被唤醒时，只有一个能获得资源
- 循环确保条件真正满足
```

**虚假唤醒（Spurious Wakeup）**：
某些操作系统实现中，条件变量可能在没有信号的情况下被唤醒。使用while循环检查条件可以正确处理这种情况。

#### 3.3.3 生产者-消费者模式

**PacketQueue 设计分析**：

```
生产者-消费者模式在播放器中的应用:

┌─────────────────────────────────────────────────────────────┐
│                      PacketQueue                            │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  链表结构: first_pkt → pkt → pkt → pkt → last_pkt   │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │  nb_packets │  │  size       │  │  SDL_mutex *mutex   │  │
│  │  (包数量)    │  │  (总字节数)  │  │  (线程互斥)         │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│                                    ┌─────────────────────┐  │
│                                    │  SDL_cond *cond     │  │
│                                    │  (条件变量)         │  │
│                                    └─────────────────────┘  │
└─────────────────────────────────────────────────────────────┘

生产者（读取线程）:
1. 从文件读取AVPacket
2. 加锁
3. 将包加入队列尾部
4. 发送条件信号（通知消费者）
5. 解锁

消费者（解码线程）:
1. 加锁
2. 检查队列是否为空
3. 如果空，等待条件变量
4. 从队列头部取出包
5. 解锁
6. 解码包

优势:
- 解耦生产者和消费者的速度差异
- 缓冲数据，平滑突发流量
- 支持阻塞和非阻塞两种模式
```

**队列大小控制**：

- 队列过大：内存占用高，延迟大
- 队列过小：容易欠载，播放卡顿
- 典型值：视频队列50-100包，音频队列100-200包

### 3.4 定时器原理

#### 3.4.1 时间获取函数

**SDL 提供的时间函数**：

```
时间函数对比:

┌─────────────────────┬─────────────────┬────────────────────────┐
│ 函数                │ 精度            │ 用途                   │
├─────────────────────┼─────────────────┼────────────────────────┤
│ SDL_GetTicks()      │ 毫秒            │ 一般定时、超时检测      │
│                     │                 │ 从SDL_Init()开始计时    │
├─────────────────────┼─────────────────┼────────────────────────┤
│ SDL_GetPerformance  │ 微秒/纳秒       │ 高精度计时、性能分析    │
│ Counter()           │ （取决于CPU）    │ 单调递增，不受系统时间   │
│                     │                 │ 调整影响               │
├─────────────────────┼─────────────────┼────────────────────────┤
│ SDL_Delay()         │ 毫秒            │ 简单延迟               │
│                     │                 │ 会释放CPU              │
└─────────────────────┴─────────────────┴────────────────────────┘

高精度计时示例:
计数器值 / 计数器频率 = 时间（秒）

例如:
计数器值 = 3,000,000,000
频率 = 3,000,000,000 Hz (3GHz CPU)
时间 = 1秒
```

**单调时钟 vs 系统时钟**：

- **SDL_GetTicks()**：基于系统时钟，可能因NTP同步而跳跃
- **SDL_GetPerformanceCounter()**：基于CPU计数器，单调递增，适合测量时间间隔

#### 3.4.2 精确延迟策略

**简单延迟 vs 精确延迟**：

```
简单延迟（SDL_Delay）:
┌────────────────────────────────────┐
│ SDL_Delay(16)  // 请求延迟16ms     │
│ 实际延迟: 15-20ms（不精确）         │
│ 原因: 操作系统调度粒度              │
└────────────────────────────────────┘

精确延迟（混合策略）:
┌────────────────────────────────────┐
│ 目标: 延迟16ms                     │
│                                    │
│ 步骤1: SDL_Delay(13)  // 粗略延迟  │
│        释放CPU，让出时间片          │
│                                    │
│ 步骤2: 忙等待剩余3ms               │
│        while (当前时间 < 目标时间)  │
│            ; // 空循环              │
│                                    │
│ 结果: 精度可达微秒级                │
│ 代价: 忙等待期间CPU占用100%         │
└────────────────────────────────────┘

适用场景:
- 音视频同步（需要微秒级精度）
- 帧率控制（60fps = 16.67ms/帧）
- 实时游戏循环
```

**操作系统调度粒度**：

- Windows：通常10-15ms
- Linux：通常1-10ms（可配置）
- macOS：通常1ms

#### 3.4.3 SDL 定时器

**单次定时器 vs 循环定时器**：

```
单次定时器:
SDL_AddTimer(1000, callback, data)
     │            │         │
     │            │         └── 传递给回调的用户数据
     │            │
     │            └── 回调函数
     │                Uint32 callback(Uint32 interval, void *param)
     │                返回值: 0=停止，非0=下次触发间隔
     │
     └── 延迟毫秒数（1000ms = 1秒）

循环定时器实现:
回调函数返回非0值:
Uint32 timer_callback(Uint32 interval, void *param) {
    // 执行定时任务
    do_something();

    // 返回相同间隔，实现循环
    return interval;  // 下次1000ms后再次触发
}

停止定时器:
SDL_RemoveTimer(timerID);
```

**定时器线程**：
SDL定时器在单独的线程中执行，因此回调函数需要注意线程安全。

### 3.5 事件系统

#### 3.5.1 事件驱动编程模型

**事件循环的核心思想**：

```
传统轮询方式（低效）:
while (running) {
    if (按键被按下) {      // 每次循环都检查
        处理按键();
    }
    if (鼠标被点击) {      // 每次循环都检查
        处理点击();
    }
    // 大量CPU浪费在空轮询上
}

事件驱动方式（高效）:
while (running) {
    SDL_WaitEvent(&event);   // 阻塞等待事件
                             // 没有事件时CPU休眠

    switch (event.type) {
        case SDL_KEYDOWN:    // 有按键事件才处理
            处理按键(event.key);
            break;
        case SDL_MOUSEBUTTONDOWN:  // 有鼠标事件才处理
            处理点击(event.button);
            break;
    }
}
```

**事件队列机制**：

```
事件产生源:                    事件队列:              应用程序:
┌─────────┐                   ┌─────────────┐       ┌─────────┐
│ 键盘    │──→ 按键事件        │ 事件1       │────→  │ 处理    │
└─────────┘                   │ 事件2       │       └─────────┘
                              │ 事件3       │
┌─────────┐                   │ ...         │
│ 鼠标    │──→ 点击事件        └─────────────┘
└─────────┘                   ↑ 先进先出（FIFO）
                              │
┌─────────┐                   │
│ 定时器  │──→ 定时事件        │
└─────────┘                   │
                              │
┌─────────┐                   │
│ 用户代码│──→ 自定义事件      │
└─────────┘                   │
                              │
事件可以:                     │
- 入队（SDL_PushEvent）       │
- 出队（SDL_PollEvent）       │
- 等待（SDL_WaitEvent）       │
```

**事件类型分类**：

- **系统事件**：窗口事件、退出事件
- **输入事件**：键盘、鼠标、触摸、手柄
- **音频事件**：音频设备变化
- **用户事件**：自定义事件（SDL_USEREVENT到SDL_LASTEVENT）

#### 3.5.2 自定义事件的应用

**在视频播放器中使用自定义事件**：

```
场景: 解码线程需要将解码后的图像传递给UI线程显示

问题:
- 解码线程不能直接在UI线程中操作控件
- 需要使用线程安全的方式通信

解决方案 - 自定义事件:

解码线程:                              UI线程:
┌─────────────────┐                   ┌─────────────────┐
│ 解码完成一帧图像 │                   │ 运行事件循环     │
│                 │                   │                 │
│ 创建自定义事件   │                   │ SDL_PollEvent() │
│ event.type =    │                   │                 │
│   SDL_USEREVENT │                   │ 收到自定义事件   │
│ event.user.data1│──────────────────→│ 调用显示函数     │
│   = 图像指针    │                   │ 更新界面        │
│                 │                   │                 │
│ SDL_PushEvent() │                   │ 继续事件循环     │
└─────────────────┘                   └─────────────────┘

优势:
- 线程安全（SDL事件队列是线程安全的）
- 解耦解码和显示
- 符合SDL的事件驱动模型
```

## 四、音视频同步原理深度解析

### 4.1 同步问题的本质

**为什么需要同步？**

```
理想情况（完全同步）:
时间轴:  0s      1s      2s      3s      4s
         │       │       │       │       │
视频:    [V0]    [V1]    [V2]    [V3]    [V4]
音频:    [A0]    [A1]    [A2]    [A3]    [A4]
         ↑       ↑       ↑       ↑       ↑
       完全对齐，声画同步

实际情况（需要同步控制）:
时间轴:  0s      1s      2s      3s      4s
         │       │       │       │       │
视频:    [V0]    [V1]    [V2]    [V3]    [V4]
音频:    [A0]        [A1]    [A2]    [A3]    [A4]
         ↑           ↑
       音频延迟0.5秒  ← 用户感受到声画不同步

原因:
- 视频解码和音频解码速度不同
- 网络传输抖动
- 系统调度延迟
- 硬件处理时间差异
```

**不同步的感知阈值**：

- 音频超前视频：可容忍范围约20-40ms
- 视频超前音频：可容忍范围约-20到-90ms
- 超过125ms：明显可感知
- 超过200ms：严重影响观看体验

### 4.2 同步策略对比

| 策略       | 原理            | 优点            | 缺点        | 适用场景   |
| -------- | ------------- | ------------- | --------- | ------ |
| **音频同步** | 以音频为基准，视频追赶音频 | 人对音频敏感，不易察觉调整 | 视频可能丢帧或重复 | 大多数播放器 |
| **视频同步** | 以视频为基准，音频追赶视频 | 画面流畅          | 音频可能变调    | 特殊场景   |
| **外部时钟** | 使用独立时钟，两者都追赶  | 精确控制          | 实现复杂      | 专业播放器  |

### 4.3 音频同步策略详解

**为什么选择音频作为基准？**

```
人耳 vs 人眼的敏感度:

音频:
- 人耳对时间变化非常敏感
- 20ms的音频延迟就能被察觉
- 音频变速（变调）很容易听出来
- 音频丢帧会产生明显的"跳音"

视频:
- 人眼对帧率变化相对不敏感
- 可以容忍偶尔的丢帧或重复帧
- 视频帧的微小时间偏移不易察觉
- 画面稍微快慢变化不影响观感

结论:
音频应该"自然播放"，视频去适应音频
```

**同步实现机制**：

```
同步流程:

视频解码线程:                    音频播放:
┌─────────────────┐             ┌─────────────────┐
│ 解码视频帧V     │             │ 音频自然播放    │
│ 获取PTS = 100ms │             │ 更新audio_clock │
│                 │             │                 │
│ 查询音频时钟    │────────────→│ 返回当前时间    │
│ audio_clock     │             │ 比如: 90ms      │
│ = 90ms          │             │                 │
│                 │             │                 │
│ 计算差异:       │             │                 │
│ diff = 100-90   │             │                 │
│ = 10ms (超前)   │             │                 │
│                 │             │                 │
│ 决策:           │             │                 │
│ 视频超前10ms    │             │                 │
│ 延迟10ms再显示  │             │                 │
│                 │             │                 │
│ SDL_Delay(10)   │             │                 │
│                 │             │                 │
│ 显示视频帧      │             │                 │
└─────────────────┘             └─────────────────┘

同步阈值控制:
- diff < -10ms: 视频落后，立即显示（或丢帧）
- -10ms < diff < 10ms: 正常范围，直接显示
- diff > 10ms: 视频超前，延迟显示
- diff > 100ms: 严重不同步，强制同步
```

**音频时钟更新**：
音频时钟通过回调函数持续更新，计算当前播放位置：

```
audio_clock = 当前播放的帧PTS + (已播放采样数 / 采样率)
```

**视频调整策略**：

1. **延迟显示**：视频超前时，等待一段时间再显示
2. **立即显示**：视频落后但在阈值内，立即显示
3. **丢帧**：视频严重落后，跳过当前帧
4. **重复帧**：视频严重超前，重复显示上一帧

## 六、核心模块详解

### 6.1 VideoPlayer 类 - 播放核心

**继承**: `QThread`

**职责**: 负责视频解码、音频解码、音视频同步、网络流处理

**多线程架构**：

```
主线程 (VideoPlayer::run)
    │
    ├── 读取线程 (主循环)
    │   └── av_read_frame() → 读取数据包 → 放入队列
    │
    ├── 视频解码线程 (video_thread)
    │   └── 从队列取包 → 解码 → 格式转换 → 发送图像信号
    │
    └── 音频回调 (audio_callback)
        └── SDL 定时调用 → 从队列取包 → 解码 → 播放
```

### 6.2 PacketQueue 类 - 数据包队列

**职责**: 线程安全的音视频数据包队列

**设计要点**：

- 使用SDL_mutex保护共享数据
- 使用SDL_cond实现阻塞等待
- 链表结构支持动态增长
- 统计包数量和总大小

### 6.3 PlayerDialog 类 - 用户界面

**继承**: `QDialog`

**职责**: 提供用户交互界面，控制播放器，显示视频画面

**主要功能**：

- **视频显示**: 使用 OpenGL 硬件加速渲染
- **进度条**: 支持鼠标拖拽跳转
- **音量控制**: 滑块调节音量
- **键盘快捷键**:
  - `空格`: 暂停/播放
  - `←`: 快退 10 秒
  - `→`: 快进 10 秒
  - `↑`: 增加音量
  - `↓`: 降低音量

## 七、关键技术实现

### 7.1 网络流媒体支持

#### 7.1.1 支持的协议

| 协议         | URL 示例                               |
| ---------- | ------------------------------------ |
| HTTP/HTTPS | `http://example.com/video.mp4`       |
| HLS        | `http://example.com/playlist.m3u8`   |
| RTMP       | `rtmp://live.example.com/app/stream` |
| RTSP       | `rtsp://192.168.1.100:554/stream`    |

#### 7.1.2 FFmpeg 网络配置要点

**基础网络设置**：

- **stimeout**: 套接字超时时间，防止无限等待
- **max_delay**: 最大延迟限制，控制缓冲大小
- **buffer_size**: 接收缓冲区大小

**HLS 专用设置**：

- **live_start_index**: 直播流起始播放位置，0表示从第一个切片开始
- **hls_allow_cache**: 是否允许缓存切片
- **seekable**: 强制启用Seek支持

**RTSP 配置**：

- **rtsp_transport**: 传输协议选择（tcp/udp）

### 7.2 Seek 功能实现

#### 7.2.1 Seek 流程

```
用户拖动进度条
    │
    ▼
seek(pos)  ← 设置 seek_pos 和 seek_req 标志
    │
    ▼
读取线程检测到 seek_req
    │
    ├── av_seek_frame()  ← 跳转到目标位置
    │
    ├── 清空音视频队列
    │
    ├── 发送 FLUSH_DATA 包 ← 通知解码器清空缓存
    │
    └── 重置时钟
    │
    ▼
解码线程收到 FLUSH_DATA
    │
    └── avcodec_flush_buffers()  ← 清理解码器缓存
```

**Seek标志说明**：

- **AVSEEK_FLAG_BACKWARD**: 向后查找，确保找到关键帧
- **AVSEEK_FLAG_BYTE**: 按字节位置Seek（某些格式支持）
- **AVSEEK_FLAG_ANY**: 允许Seek到非关键帧（需要解码器支持）
- **AVSEEK_FLAG_FRAME**: 按帧号Seek

### 7.3 HLS 直播流优化

#### 7.3.1 问题与解决方案

| 问题      | 解决方案                 | 原理                     |
| ------- | -------------------- | ---------------------- |
| 无法从头播放  | `live_start_index=0` | HLS默认播放最新切片，设置为0从第一个开始 |
| 无法 Seek | `seekable=1`         | 强制FFmpeg启用Seek支持       |
| 播放卡顿    | 增大缓冲区                | 网络抖动时提供数据缓冲            |
| 延迟过高    | `max_delay=500000`   | 限制最大缓冲延迟               |
